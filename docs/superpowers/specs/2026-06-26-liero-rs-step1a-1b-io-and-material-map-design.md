# Step 1a+1b — IO primitives + level material map

Status: **draft for review** · 2026-06-26
Part of: `2026-06-26-liero-rs-step1-data-formats-overview.md`

## Purpose

First slice of step 1. Build a Rust loader that reads a `.lev` file and produces
`LevelData { width, height, material_id }` **byte-identical** to what C++
`Level::load` parses. This is the only level data the simulation (step 2) needs;
palette, display layers, sprites and sounds are deferred to later slices.

Delivers: a new `assets` crate with an `io` module (1a) and a `level` module (1b),
plus a golden differential test proving the Rust output matches C++.

## Scope

**Modernization note (per the project charter):** we do **not** port C++'s
`io::Reader`/`MemReader`/`coding.hpp`. The *behaviour* (which bytes produce which
material map) is locked and differential-tested; the *implementation* is
idiomatic Rust — `std::io`/byte-slice parsing, `from_le_bytes`, and `Result`-based
errors. So the "IO primitives" of slice 1a collapse into ordinary std usage rather
than a custom reader abstraction.

**Included:**
1. `assets` crate (Bevy-free). Idiomatic parsing helpers as needed (e.g. a small
   cursor over `&[u8]`, `u16::from_le_bytes`) — **not** a reimplementation of the
   C++ reader/coding layer.
2. `level.rs`: `LevelData` + `load(bytes: &[u8]) -> Result<LevelData, LevelError>`
   handling the two formats the engine reads for the material map:
   - **Legacy:** headerless 504×350 → 176400 raw material bytes.
   - **Sized (`OLLEVEL2`):** magic + version + `u16` width + `u16` height + w×h
     material bytes.
   - For files with trailing `POWERLEVEL`/`MODERNLV` blocks (e.g. the shipped
     `modern_test.lev`): read the header + material bytes and **ignore** the rest.
3. C++ oracle dumper (CMake-gated, links `game`) that loads the three inputs via
   the real `Level::load` and dumps `width height` + material bytes.
4. Rust golden test reproducing all three.

**NOT included** (later slices): palette parsing, MODERNLV display/animation,
per-pixel `materials` derivation (needs the TC material table — slice 1e),
sprites, sounds, deflate (the material map is uncompressed).

`LevelData` and a small error type:
```rust
pub struct LevelData {
    pub width: i32,
    pub height: i32,
    pub material_id: Vec<u8>, // width*height palette indices, row-major
}

pub enum LevelError { Truncated, BadDimensions(i32, i32) } // idiomatic Result errors
```

Parsing uses idiomatic Rust over the input slice: detect the `OLLEVEL2` magic by
comparing the first 8 bytes, read `width`/`height` with `u16::from_le_bytes`, and
take the `width*height` material bytes as a `Vec<u8>`. No custom `Reader` trait.
Invalid input returns `Err(LevelError)` rather than C++'s `bool` return.

## Semantics that MUST be preserved

| C++ | Rust | Trap |
|---|---|---|
| `OLLEVEL2` magic (8 bytes) | exact byte compare | match the exact magic string |
| width/height `u16` little-endian | `read_u16_le` | endianness; `level.cpp:247-248` |
| legacy = 504×350 headerless | constants | only when no/short magic + exactly 176400 bytes available |
| material bytes = raw `u8`, row-major | `Vec<u8>` | no transformation; palette indices 0–255 |
| dimension cap 4096×4096, 0 rejected | same validation | `level.cpp:232`; reject invalid like C++ does |
| trailing blocks ignored | stop after w×h material bytes | do not fail on extra bytes |

The exact format-detection logic (how C++ decides legacy vs OLLEVEL2 from the
first bytes) is read from `level.cpp:229+` during the plan and mirrored exactly.

## Differential test harness

Same oracle pattern as step 0, but the dumper links the engine. The C++
`level_dump` builds three `LevelData` via the **real** `Level::load`:

1. The shipped `data/TC/openliero/Levels/modern_test.lev`.
2. A synthetic **legacy** buffer (504×350, a deterministic fill pattern).
3. A synthetic **OLLEVEL2** buffer (small, e.g. 8×4, a deterministic pattern).

For each it writes a record to `golden/level.txt`:
```
<width> <height>
<material_id byte 0> <byte 1> … (decimal, space- or newline-separated)
```
The Rust test builds the same two synthetic buffers (identical fill patterns,
defined verbatim in both languages), loads the real file from disk, and asserts
its `LevelData` matches the golden line-for-line.

**Why link `game`:** `Level::load` depends on `Material`, `io`, and `Settings`.
Rather than hand-pick `.cpp` files, the dumper links the existing `game` target
(behind `OPENLIERO_BUILD_ORACLE_DUMP=ON`). `gen_golden.sh` is extended to build
this target via CMake and run it. The step-0 math dumper stays as-is.

*Open consideration:* `Level::load` may require a minimal `Settings`/material
table to run even though we only read `material_id`. The plan determines the
minimal scaffolding by reading `level.cpp` and constructs it in the dumper.

## Definition of done

- [ ] `assets` crate compiles, no dependencies.
- [ ] `cargo test` green: `level_golden` matches C++ for all three inputs.
- [ ] Real `modern_test.lev` round-trips (width/height + every material byte).
- [ ] Both synthetic formats (legacy + OLLEVEL2) match.
- [ ] Oracle dumper is a committed CMake target; `gen_golden.sh` regenerates
      `level.txt` reproducibly.
- [ ] CI (`rust.yml`) runs the new test (it already runs `cargo test --workspace`).

## Open questions (decide during the plan)

1. **Dumper scaffolding:** exactly what `Level::load` needs to run headless
   (Settings fields, material table). Read `level.cpp` and provide the minimum.
2. **Format detection:** confirm the precise legacy-vs-OLLEVEL2 decision in
   `level.cpp` and mirror it (magic probe + size fallback).
3. **Golden size:** `modern_test.lev` material map could be large; if so, store
   material bytes compactly (e.g. one row per line, or a hash + a sample). Decide
   when the file size is known. *Default:* full bytes, newline-wrapped per row.
