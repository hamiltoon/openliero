# Read by BOTH the C++ dumper (oracle_dump_sim_physics, slice-4a object-loops +
# `weapon` directive) and the Rust N-tick test. Same grammar as slice 3 plus the
# `weapon <slot> <name>` directive.
#
# Control bits (7-bit ControlState, see worm.hpp Control enum):
#   Up=1  Down=2  Left=4  Right=8  Fire=16  Change=32  Jump=64
#
# Aiming convention (cossin_table indexed 0..127): 0=up, 32=right, 64=down,
# 96=left. For this worm the resting/settled aim points down-left into the dirt it
# stands on, so:
#   * Up(1)   raises the gun toward the sky  -> a fired shot flies UP and away.
#   * Down(2) lowers the gun into the floor  -> a fired shot drives DOWN into dirt.
# ProcessAiming accelerates slowly (AimAcc 4000, cap 70000), so a meaningful swing
# needs ~30 held ticks — that is why the up-aim phase is long.
#
# THE FAN ("explodes into nothing"): parts=1, dirt_effect=-1, create_on_exp=-1,
# splinter=0, gravity=0, expl_ground=true, time_to_explo=45, time_to_explo_v=10,
# delay=0, loading_time=220 (ammo 150, so loading_left stays 0 => fire-ready every
# tick). `weapon 0 FAN` puts the fan in slot 0; the dumper hardcodes
# current_weapon=0, so slot 0 is the only fire-ready slot.
#
# HARD CONSTRAINTS (the implementer verified every input line below obeys these):
#   * NEVER set Left(4) and Right(8) together on the same worm/tick.
#   * worm1 starts visible 0 -> Worm::Process is gated `if (visible)` (worm.cpp:218),
#     so worm1 NEVER moves, fires, or draws RNG: its component hash is CONSTANT and
#     the (omitted) worm-hit loop can never apply. The no-shot-within-detect_distance
#     -of-a-visible-worm rule is therefore trivially satisfied (fan detect_distance=1,
#     worm1 invisible).
#   * worm0 stays health 100, in-bounds, grounded; it only ever sets Up/Down/Fire.
#   * dirt_effect=-1 => NEITHER shot digs, not even the ground-hit one. The `level`
#     column is CONSTANT on every line: that pristine terrain is the 4a guarantee,
#     not an accident. If `level` ever moves a dig/dirt-effect leaked in — fix it.
